# Paddle App2Web Guide

This guide walks through the full setup for launching Paddle App2Web with Helium.

App2Web is not a separate replacement for normal Helium onboarding. It is a web checkout path layered on top of a working Helium app integration and paywall setup.

Use this guide when you want a mobile user to see a Helium paywall in your app, complete checkout through Paddle in a browser, return to the app, and receive the correct entitlement.

## What you are building

The App2Web flow has five parts:

1. Your app shows an in-app Helium paywall.
2. The user taps a Paddle checkout CTA.
3. Helium opens a hosted web paywall and Paddle checkout in the browser.
4. Paddle redirects the user back into your app through a success or cancel URL.
5. Helium refreshes entitlement state so the app can unlock the user or keep them unentitled.

The key idea: the user starts in the app, pays on the web, then returns to the app.

| Term | Meaning |
| :-- | :-- |
| In-app paywall | The Helium paywall shown inside your app. This is where the user starts. |
| Hosted web paywall | The browser-based Helium paywall that backs the Paddle checkout flow. |
| Paddle product | The item being sold in Paddle, such as a monthly or annual subscription. |
| Paddle price | The billing configuration for a product, such as monthly, annual, trial, or no-trial. |
| Success URL | The app link Paddle uses after a completed purchase. |
| Cancel URL | The app link Paddle uses when checkout does not complete. |
| Entitlement | The app's understanding that a user should have access to paid features. |

## Before you start

You should have a normal Helium launch path in place before App2Web becomes the production purchase path.

You need:

- The Helium SDK integrated in your app.
- Your normal in-app purchase products configured.
- A baseline Helium paywall or paywall plan.
- Admin access to the Helium dashboard.
- A Paddle account.
- An app deep link or universal link that opens your app reliably.
- A plan for user identity and entitlement checks.
- RevenueCat access, if your app uses RevenueCat.

<Tip>
  Start Paddle account approval and app SDK work early. You can configure most of the system and test with Paddle sandbox before live approval. You cannot accept real live payments until Paddle approves your live account.
</Tip>

## Recommended setup order

You can configure most of Paddle App2Web before you actually show it to users. That is usually the best path.

There is no downside to getting the SDK and Paddle configuration done early:

- Adding SDK support does not make Paddle live.
- Connecting Paddle does not force users into web checkout.
- You can fix the configuration later if testing exposes an issue.
- Once setup is complete, you can turn App2Web on whenever the paywall, targeting, and QA are ready.

Recommended order:

1. Start Paddle account approval.
2. Add SDK support for external web checkout.
3. Configure success and cancel return URLs.
4. Decide how Helium should identify users and grant entitlements.
5. Connect Paddle sandbox to Helium.
6. Sync products, prices, trials, and discounts.
7. Build the in-app App2Web paywall and linked hosted web paywall.
8. Test with a test audience.
9. Move to live Paddle credentials after approval.
10. Launch through controlled targeting or an experiment.

## Add Paddle support to the SDK

This is one of the most important sections. It is usually worth adding SDK support early because it does not make anything live by itself. It only gives your app the ability to send users to web checkout and receive them when they come back.

### What the SDK setup does

When a user taps a Paddle purchase CTA:

1. Helium opens the browser checkout flow.
2. The user completes checkout, cancels, or fails payment.
3. Paddle sends the browser to a URL that opens your app.
4. Your app passes that URL back to Helium.
5. Helium refreshes entitlement state.

In plain English: your app needs to know how to leave for browser checkout and how to receive the user when they come back.

### Deep links and return URLs

There are two different kinds of links that often get confused:

| Link type | What it is for |
| :-- | :-- |
| Helium preview or test deep links | Used to open or debug Helium paywalls from the dashboard during development. |
| Paddle success and cancel URLs | Used after browser checkout to return the user to your app. |

Both may use your app's custom scheme or universal links, but they solve different problems.

The success URL and cancel URL are not pages the user needs to read. They are return doors back into your app. Use URLs that open the app reliably, then pass the opened URL to Helium so the SDK can finish the checkout handoff.

Recommended pattern:

- Use app links that clearly represent the Paddle return path, such as `yourapp://helium/paddle/success` and `yourapp://helium/paddle/cancel`.
- Make sure both links open the app in every app state: cold start, backgrounded, and already open.
- Pass both returned URLs to Helium's URL handler.

<Note>
  If your app already has one reliable app-open deep link, you may be able to reuse it for both success and cancel. Confirm this with Helium before launch. Separate success and cancel paths are usually easier to debug.
</Note>

### Success URL vs cancel URL

`successURL` is where Paddle sends the user after a completed purchase.

`cancelURL` is where Paddle sends the user when checkout does not complete. This can include explicit cancellation or a failed payment, depending on the checkout state.

Your app should treat the return URL as a signal to refresh state, not as proof by itself that the purchase succeeded.

- On success return: refresh entitlements and unlock only if the entitlement is active.
- On cancel return: refresh entitlements and keep the user unentitled unless Helium says they now have access.
- On payment failure: do not unlock the user.
- On abandoned checkout: do not unlock the user.

<Warning>
  Never unlock paid access just because the app received a success URL. Always check entitlement state after the user returns.
</Warning>

### User identity and entitlements

Helium grants Paddle entitlements based on the user ID you provide and, where available, Apple ID purchase context. Your setup should make it clear which identity Helium should trust before and after purchase.

Before launch, choose the path that matches your app.

| Path | What to do |
| :-- | :-- |
| The user has a stable user ID before purchase | This is the happy path. Set the user ID before the user sees a paywall, keep it stable across app launches, and check Helium entitlements after checkout returns to the app. |
| The user may purchase before sign-in, but you learn a stable user ID later | Enable Paddle entitlement without a user ID, then set the stable user ID as soon as the user signs in. Test anonymous purchase, login-after-purchase, reinstall, and account switching before launch. |
| The user ID may change after purchase | Update Helium when the stable user ID changes. Helium should reconcile the purchase from the earlier identity to the later stable identity, but you should test this exact path before launch. |
| The app never has a stable user ID | This is not recommended. Helium can still use device and Apple ID context where available, but cross-device behavior, account recovery, RevenueCat mapping, and support debugging will be weaker. |

Rules of thumb:

- Set a stable user ID before purchase whenever possible.
- Do not use email addresses, IDFA, hardcoded strings, or temporary session IDs as the primary user ID.
- If you allow purchase before sign-in, configure the SDK/dashboard path that allows entitlement without a user ID before testing App2Web.
- If the user later signs in, set the stable user ID immediately.
- After checkout returns to the app, check Helium entitlement state. Do not unlock based only on the return URL.
- If the same person may appear under multiple IDs, test that the entitlement follows the right user.

Think of your stable user ID as the most controllable entitlement anchor. Apple ID context can help where available, but if both the user ID and Apple ID context change, treat that as an account recovery edge case and make sure your support path can resolve it.

<Warning>
  If your app supports anonymous purchase, login-after-purchase, logout, account switching, or no stable user ID at all, entitlement QA is a required launch gate.
</Warning>

### Why configure SDK support early

Getting the SDK configuration done early does not make App2Web live. Users only see the Paddle path after your dashboard paywall, targeting, and workflow setup show it.

Early SDK setup is useful because it means:

- You can ship the app build while Paddle approval is still in progress.
- You can test deep links, return handling, and entitlement refresh before launch.
- You can fix SDK issues without waiting until the day you want to turn App2Web on.
- You can launch App2Web later from Helium targeting or experiments without needing another app release.

If testing reveals an issue, fix it before enabling App2Web for real users.

## Connect Paddle to Helium

After the SDK path is underway, connect Paddle to Helium so products can be synced and used in paywalls.

<Steps>
  <Step title="Create a Paddle API key">
    Create an API key in Paddle with the permissions Helium requires to read products, read prices, create transactions, manage subscriptions, read discounts, configure notifications, and create customer portal sessions.

    Never paste API keys into chat, email, client-side code, or a public repository.

    <Accordion title="Required Paddle API key permissions">
      | Permission | Why Helium needs it |
      | :-- | :-- |
      | `product.write` | Create and sync products. |
      | `price.write` | Create and sync prices. |
      | `customer.write` | Create customers for end users. |
      | `transaction.write` | Create transactions for purchases. |
      | `subscription.write` | Manage subscription lifecycle. |
      | `adjustment.read` | Read refunds and chargebacks. |
      | `notification_setting.write` | Configure notifications or webhooks. |
      | `payment_method.read` | Read saved payment methods where needed. |
      | `client_token.write` | Create client-side tokens. |
      | `discount.read` | Read discount information for paywalls. |
      | `customer_portal_session.write` | Create customer portal sessions for subscription management. |
    </Accordion>
  </Step>
  <Step title="Add the Paddle API key to Helium">
    In the Helium dashboard, go to Settings, then Integrations, then Paddle. Paste the API key into the Paddle integration.
  </Step>
  <Step title="Configure Paddle notifications">
    Helium will provide a notification or webhook destination. Add it in Paddle, then copy the signing secret back into Helium if prompted.
  </Step>
  <Step title="Sync products and prices">
    Sync Paddle products into Helium after setup. Sync again after product, price, trial, or discount changes.
  </Step>
  <Step title="Repeat for live">
    Sandbox and live Paddle accounts are separate. When moving to production, configure live credentials, live products, live prices, and live notifications.
  </Step>
</Steps>

<Warning>
  Sandbox purchases prove that the integration shape works. They do not prove that your live Paddle account, live products, live notifications, or live approval are ready.
</Warning>

## Configure products, trials, and discounts

Paddle catalog setup matters because Helium uses the product and price structure to decide what to show and what entitlement to grant.

### Products and prices

Use simple, obvious naming. A future teammate should be able to tell what each product and price is without opening five dashboards.

Recommended naming:

- App name or plan name.
- Billing period.
- Environment, if useful.
- Trial or no-trial state, if applicable.

Examples:

- `Pro Monthly - Paddle - No Trial`
- `Pro Monthly - Paddle - 7 Day Trial`
- `Pro Annual - Paddle - No Trial`

### Trials

Helium's recommended trial model is trial/no-trial pairing.

For each offer that needs a trial:

1. Create a trial Paddle price or product.
2. Create an equivalent no-trial Paddle price or product.
3. Link the trial and no-trial versions in Helium.
4. Let Helium route eligible users to the trial version and ineligible users to the no-trial version.

Helium currently treats Paddle trials and discounts as new-user acquisition offers. In practice, that means a user should only get a given trial or discount once.

<Warning>
  Do not rely on a raw Paddle trial alone unless Helium has confirmed that this is the intended behavior for your launch. A raw trial can behave differently from Helium's trial eligibility logic.
</Warning>

### Discounts

If your launch uses discounts, confirm the discount model with Helium before building the paywall.

Current guidance:

- Discounts are treated as acquisition offers.
- A customer should only receive the acquisition discount once.
- More flexible offer variations are planned, but should not be assumed for launch.

### Checkout restrictions

Some checkout rules are launch decisions, not just dashboard configuration.

Confirm whether your App2Web paywall should:

- Block prepaid cards.
- Restrict checkout to specific storefronts or regions.
- Require a specific payment method.
- Hide Paddle from users who should stay on in-app purchase.

If Helium exposes these as paywall-level settings, configure them per paywall rather than assuming one global rule fits every launch.

## Build the App2Web paywall

This is the other high-risk section. The most common mistake is treating the hosted web paywall as a normal paywall. It is not. It exists to back the browser checkout flow.

### The two-paywall model

Paddle App2Web requires two paywalls:

| Paywall | Where it appears | What it does |
| :-- | :-- | :-- |
| In-app App2Web paywall | Inside your app | Shows the user the offer and sends eligible users toward Paddle checkout. |
| Hosted web paywall | In the browser | Presents the web checkout destination and Paddle purchase flow. |

The in-app paywall is targeted through workflows, experiments, and audiences.

The hosted web paywall should not be targeted directly. It should be linked from the in-app paywall.

### Start from the approved baseline

Start from a paywall that already works with normal in-app purchases.

Before converting it to App2Web, confirm:

- The baseline paywall is approved.
- Product display and pricing are correct.
- Trial behavior is understood.
- The normal in-app purchase path has passed QA.
- Fallback behavior has been considered for the app.

App2Web should be a controlled adaptation, not a brand-new paywall that changes every variable at once.

### Product bucket rules

Keep purchase paths clean.

General rules:

- The Paddle hosted web paywall should contain Paddle products only.
- Do not put iOS or Android products on the hosted web paywall.
- Do not mix Paddle and Stripe products on the same hosted web paywall.
- If the in-app App2Web paywall includes an iOS fallback path, keep the iOS products in the continue-in-app path.
- Avoid Android products in an iOS App2Web paywall.

If your current product bucket combines iOS and Android products, create an iOS-only version for the App2Web path.

<Accordion title="How to think about duplicating a product bucket">
  A normal paywall may use one product bucket for multiple platforms. App2Web is more sensitive because the user can choose between a web purchase path and an in-app purchase path.

  For a clean App2Web setup:

  1. Keep the Paddle product in a Paddle-only bucket.
  2. Keep the iOS fallback product in an iOS-only bucket.
  3. Do not send Android products into the iOS App2Web flow.
  4. Use clear names so it is obvious which bucket belongs to which path.

  Confirm the exact dashboard steps with Helium if your current bucket structure is complex.
</Accordion>

### Build the in-app App2Web paywall

The in-app paywall should be based on the approved IAP paywall, with the purchase decision adapted for App2Web.

Recommended structure:

1. First screen: match the approved baseline paywall as closely as possible.
2. App2Web decision step: send eligible users to Paddle checkout.
3. Secondary option: provide a smaller continue-in-app path for the normal IAP purchase flow when appropriate.

Checklist:

- Primary CTA opens the Paddle web checkout path.
- Secondary CTA says something clear like `or continue in-app`.
- Continue-in-app uses the normal IAP path.
- Pricing, trial copy, plan names, and billing periods match actual Paddle configuration.
- The paywall does not imply that checkout is complete before the user pays.
- The user can recover if browser checkout is canceled or fails.

<Tip>
  If the existing paywall is a one-screen IAP paywall, consider adding a lightweight second step before checkout. This gives the user context before leaving the app for the browser.
</Tip>

### Build the hosted web paywall

The hosted web paywall is the browser destination for Paddle checkout.

Checklist:

- It contains Paddle products only.
- It is linked to the in-app App2Web paywall.
- It is published before end-to-end testing.
- The CTA starts Paddle checkout.
- Copy and pricing match the in-app paywall.
- The layout works in a mobile browser.
- Dismissal behavior is intentional. Avoid an X button if it makes the flow feel like a modal that can be safely closed without consequence; use a back affordance when that is clearer.
- Success return opens the app.
- Cancel or failure return opens the app and does not unlock the user.
- Entitlement refresh has been tested after return.

Limitations to keep in mind:

- Hosted web paywalls are less capable than in-app Helium paywalls.
- App-only variables, payloads, and localized app context may not be available after the browser handoff.
- Targeting and audience logic should happen before the user is sent to the hosted web paywall.
- Do not rely on the hosted web paywall as a standalone acquisition page unless Helium has explicitly confirmed that flow.

### Do not target the hosted web paywall directly

Add the in-app App2Web paywall to workflows, experiments, and audiences.

Do not add the hosted web paywall directly to a workflow, experiment, or targeting rule. The hosted web paywall should be reached through the linked App2Web flow.

## Configure targeting and rollout

App2Web should launch as a controlled rollout, not as a surprise global change.

Recommended rollout:

1. Create a test audience.
2. Show App2Web only to test users.
3. Run sandbox and device QA.
4. Run production feature-flag QA if needed.
5. Launch as an experiment or controlled rollout.
6. Monitor conversion, checkout completion, entitlement grants, and support tickets.

Targeting should make sure:

- Only eligible users see Paddle App2Web.
- Users outside the intended storefronts or regions do not see the Paddle CTA.
- Test users can access the flow before the public rollout.
- Existing subscribers do not see an irrelevant purchase prompt.
- Users who should stay on IAP can still use the normal in-app path.

<Warning>
  External purchase rules differ by region and app category. Confirm the supported storefronts and legal/App Review position for your launch before exposing App2Web broadly.
</Warning>

## RevenueCat setup

If your app uses RevenueCat, make sure Paddle purchases are associated with the same RevenueCat App User ID that your app expects.

RevenueCat has its own customer identity called App User ID. Helium, Paddle, and RevenueCat need to agree on which user made the purchase.

### Recommended RevenueCat configuration

Follow RevenueCat's Paddle documentation and confirm the exact setup with Helium for your app.

Typical setup:

1. Open RevenueCat.
2. Go to your project.
3. Add a Paddle web configuration.
4. Add the required Paddle API key or connection details.
5. Use automatic purchase tracking unless Helium tells you otherwise.
6. Configure App User ID matching with a custom field, typically `rc_user_id`.
7. Import Paddle products into RevenueCat.
8. Attach Paddle products to the correct RevenueCat entitlement.

Useful RevenueCat references:

- [RevenueCat Paddle Billing](https://www.revenuecat.com/docs/web/integrations/paddle)
- [RevenueCat Paddle app-to-web purchases](https://www.revenuecat.com/docs/web/guides/paddle-app-to-web)
- [RevenueCat Paddle server notifications](https://www.revenuecat.com/docs/platform-resources/server-notifications/paddle-server-notifications)

### What to test with RevenueCat

Test these before launch:

- A Paddle purchase unlocks the user in Helium immediately.
- The same purchase appears in RevenueCat under the expected App User ID.
- The correct RevenueCat entitlement becomes active.
- Anonymous users, if supported, do not create broken or duplicate entitlement state.
- Login-after-purchase attaches the purchase to the correct user.
- Logout and account switching do not leak entitlements between users.
- Restore behavior is understood and documented for your support team.

<Note>
  For the fastest post-purchase app experience, check Helium entitlement state immediately after the user returns from Paddle. RevenueCat may be slightly delayed because it depends on server notifications and identity matching.
</Note>

## Test the flow

Testing should prove the whole path, not just one dashboard screen.

| Scenario | Expected result |
| :-- | :-- |
| In-app paywall opens | The App2Web paywall appears only for the intended test audience. |
| Primary CTA opens browser | The user leaves the app and reaches the hosted web paywall or Paddle checkout. |
| Hosted web paywall loads | Copy, products, prices, and CTA are correct on mobile web. |
| Sandbox purchase succeeds | User returns to the app and Helium entitlement is active. |
| Live test purchase succeeds | Live Paddle payment works after account approval. |
| Cancel returns to app | User returns through cancel URL and remains unentitled. |
| Failed payment does not unlock | User is not granted access. |
| Abandoned checkout does not unlock | User is not granted access. |
| Continue-in-app works | User can still purchase through IAP when that path is offered. |
| Trial-eligible user sees trial | Eligible user sees the trial offer. |
| Trial-ineligible user sees no-trial offer | Ineligible user does not get another trial. |
| RevenueCat entitlement updates | RevenueCat shows the right entitlement for the right App User ID. |
| Manage subscription opens | Paddle subscriber reaches Paddle customer portal, not App Store subscription settings. |
| Existing subscriber state works | Existing subscribers do not see a misleading purchase prompt. |
| Webhook delivery succeeds | Paddle notifications reach Helium and return successful responses. |
| Rollout targeting works | Users outside the intended audience do not see App2Web. |

### Testing before live approval

You can test these before live Paddle approval:

- SDK return handling.
- Deep link behavior.
- Sandbox checkout.
- Product and price sync from sandbox.
- Hosted web paywall layout.
- Basic entitlement handoff in sandbox.
- RevenueCat mapping in a sandbox-style setup if configured.

### Testing after live approval

You need live approval for:

- Real live Paddle payments.
- Live webhook validation.
- Live product and price validation.
- Final production monitoring.
- Final support and cancellation flows.

## Compliance checklist

This section is a launch checklist, not legal advice. Confirm final requirements with your legal and App Review team.

### Apple checklist

- Confirm which storefronts and regions can see App2Web.
- Confirm the app category and external purchase rules that apply to your app.
- Confirm the app still behaves correctly for users who should use IAP.
- Do not show external purchase CTAs in unsupported storefronts.
- Do not mislead users about price, billing period, trial terms, or cancellation.

## Post-launch monitoring

Watch the first 24 hours closely.

Important signals:

- App2Web paywall opens.
- CTA press to browser open rate.
- Hosted web paywall load rate.
- Paddle checkout start rate.
- Paddle checkout completion rate.
- Cancel and failure return rate.
- Deep link return failures.
- Entitlement grant delay or failure.
- RevenueCat sync lag.
- Paddle webhook failures.
- Support tickets about access, payment failure, cancellation, trial eligibility, or subscription management.

If something looks wrong, first protect users:

1. Pause the App2Web experiment or audience.
2. Keep the normal IAP path available when possible.
3. Check Paddle notifications and Helium entitlement state.
4. Check whether the issue is checkout, deep link return, entitlement mapping, RevenueCat sync, or targeting.

## Troubleshooting

| Problem | Likely cause | First check |
| :-- | :-- | :-- |
| User never returns to app | Deep link or universal link is not configured correctly | Open the success and cancel URLs directly on a device. |
| Purchase succeeds but app stays locked | Entitlement refresh or identity mapping issue | Check Helium entitlement state for the user ID. |
| RevenueCat does not show purchase | App User ID matching or Paddle notification issue | Check `rc_user_id`, RevenueCat Paddle config, and server notifications. |
| User sees Paddle when they should not | Targeting/audience issue | Check workflow, experiment, and audience rules. |
| User sees iOS product in web path | Product bucket setup issue | Confirm hosted web paywall uses Paddle-only products. |
| Hosted web paywall cannot publish | Missing or invalid product/linking state | Confirm the web paywall has valid Paddle products and is linked correctly. |
| Trial appears for an ineligible user | Trial/no-trial mapping issue | Confirm Helium trial pairing and eligibility logic. |
| Sandbox works but live fails | Live account/catalog/credential mismatch | Confirm live approval, live products, live API key, and live notifications. |
| Manage subscription opens wrong destination | App is routing Paddle subscribers to App Store settings | Use the Paddle portal session for Paddle subscribers. |
| Checkout fails for some users | Payment method, region, compliance, or Paddle account issue | Check Paddle transaction details and eligibility targeting. |
