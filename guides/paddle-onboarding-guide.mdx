title: "Paddle App2Web"
description: "Implement Helium's Paddle App2Web checkout flow without tanking conversion."
---

Helium's Paddle App2Web flow lets teams customize and experiment with web checkout while preserving the in-app purchase path on iOS.

Apps often see 30-40% higher proceeds when eligible purchases move to web checkout. Implement the first flow in this order: SDK support, Paddle setup, paywall conversion, entitlement handling, and launch.

## What you are building

The App2Web flow has five parts:

1. Your app shows an in-app Helium paywall.
2. The user taps a Paddle checkout CTA.
3. Helium opens a hosted web paywall and Paddle checkout in the browser.
4. After a successful purchase or when the user closes checkout, Helium redirects the user back into your app.
5. Helium refreshes entitlement state so the app can unlock access or keep the user unentitled.

The key idea: the user starts in the app, pays on the web, then returns to the app.

| Term | Meaning |
| :-- | :-- |
| In-app paywall | The Helium paywall shown inside your app. This is where the user starts. |
| Hosted web paywall | The browser-based Helium paywall that backs the Paddle checkout flow. |
| Paddle product | The item being sold in Paddle, such as a monthly or annual subscription. |
| Paddle price | The billing configuration for a product, such as monthly, annual, trial, or no-trial. |
| Success URL | A link Paddle uses after a completed purchase. |
| Cancel URL | A link Paddle uses when checkout does not complete. |
| Entitlement | The access state that says a user should have paid features. |

## Before you start

Before configuring App2Web with Paddle, have a working iOS in-app purchase flow through Helium. App2Web is an additional purchase path for eligible iOS users, not a replacement for your normal IAP setup.

Helium App2Web with Paddle currently supports iOS only.

Prerequisites:

- The Helium SDK integrated in your iOS app.
- Working iOS in-app purchase products in Helium.
- A baseline Helium paywall that can already sell through IAP.
- Admin access to the Helium dashboard.
- A Paddle account.
- An app deep link or universal link that opens your app reliably.
- A plan for user identity and entitlement checks.
- RevenueCat access, if your app uses RevenueCat.

<Tip>
  Start Paddle account review and app SDK work early. You can configure the Helium SDK, paywalls, products, and targeting while Paddle review is in progress. Live payments require an approved Paddle account.
</Tip>

## Add Paddle support to the SDK

Add this alongside the standard Helium SDK integration. This does not make Paddle visible by itself; users only see Paddle after the Helium paywall and targeting are configured.

<Tabs>
  <Tab title="iOS native">
    <Note>
      Paddle App2Web requires Helium iOS SDK 4.4.7 or later. If you have not integrated Helium yet, start with the [iOS SDK Quickstart](/sdk/quickstart-ios).
    </Note>

    ```swift
    // Set the stable user ID before showing the paywall, if available.
    Helium.identify.userId = "user-123"

    // Enable Paddle checkout before initializing Helium.
    Helium.config.enableExternalWebCheckout(
      successURL: "yourapp://helium/paddle/success",
      cancelURL: "yourapp://helium/paddle/cancel",
      paymentProcessors: .paddle
    )

    // Only set this if users can buy before signing in.
    Helium.config.allowWebCheckoutWithoutUserId = true

    // Pass returned checkout URLs to Helium.
    func application(
        _ app: UIApplication,
        open url: URL,
        options: [UIApplication.OpenURLOptionsKey : Any] = [:]
    ) -> Bool {
        Helium.shared.handleURL(url)
        return true
    }

    // If checkout happened before sign-in, set the stable user ID after sign-in.
    Helium.identify.userId = "user-123"

    // Check Helium entitlement state after checkout returns.
    if await Helium.entitlements.hasActivePaddleEntitlement() {
        // Unlock paid access.
    }

    // Route Paddle subscribers to Paddle customer portal.
    do {
        let url = try await Helium.shared.createPaddlePortalSession()
        UIApplication.shared.open(url)
    } catch {
        print("Failed to open Paddle customer portal: \(error)")
    }
    ```

    If your app uses SwiftUI lifecycle APIs, call `Helium.shared.handleURL(url)` from `onOpenURL`.
  </Tab>

  <Tab title="React Native / Expo">
    <Note>
      Paddle App2Web requires expo-helium 3.4.4 or later. If you have not integrated Helium yet, start with the [React Native SDK Quickstart](/sdk/quickstart-react-native).
    </Note>

    ```tsx
    import { useEffect } from "react";
    import * as Linking from "expo-linking";
    import {
      createPaddlePortalSession,
      enableExternalWebCheckout,
      hasActivePaddleEntitlement,
      heliumHandleURL,
      setCustomUserId,
    } from "expo-helium";

    // Set the stable user ID before showing the paywall, if available.
    setCustomUserId("user-123");

    // Enable Paddle checkout before initializing Helium.
    enableExternalWebCheckout(
      successURL: "yourapp://helium/paddle/success",
      cancelURL: "yourapp://helium/paddle/cancel",
      paymentProcessors: ["paddle"],
    );

    // Pass returned checkout URLs to Helium.
    useEffect(() => {
      const subscription = Linking.addEventListener("url", (event) => {
        heliumHandleURL(event.url);
      });

      void Linking.getInitialURL().then((url) => {
        if (url) heliumHandleURL(url);
      });

      return () => subscription.remove();
    }, []);

    // Check Helium entitlement state after checkout returns.
    if (await hasActivePaddleEntitlement()) {
      // Unlock paid access.
    }

    // Route Paddle subscribers to Paddle customer portal.
    const url = await createPaddlePortalSession();
    if (url) {
      await Linking.openURL(url);
    }
    ```
  </Tab>

  <Tab title="Flutter">
    <Note>
      Paddle App2Web requires helium_flutter 3.3.4 or later. If you have not integrated Helium yet, start with the [Flutter SDK Quickstart](/sdk/quickstart-flutter).
    </Note>

    ```dart
    final heliumFlutter = HeliumFlutter();

    // Enable Paddle checkout before initializing Helium.
    await heliumFlutter.enableExternalWebCheckout(
      successURL: "yourapp://helium/paddle/success",
      cancelURL: "yourapp://helium/paddle/cancel",
      paymentProcessors: {HeliumWebCheckoutProcessor.paddle},
    );

    // Set the stable user ID during initialization, if available.
    await heliumFlutter.initialize(
      apiKey: "<your-helium-api-key>",
      customUserId: "user-123",
    );

    // Pass returned checkout URLs to Helium.
    _linkSubscription = AppLinks().uriLinkStream.listen((uri) {
      heliumFlutter.handleURL(uri.toString());
    });

    // Check Helium entitlement state after checkout returns.
    final hasPaddle = await heliumFlutter.hasActivePaddleEntitlement();
    if (hasPaddle) {
      // Unlock paid access.
    }

    // Route Paddle subscribers to Paddle customer portal.
    final portalUrl = await heliumFlutter.createPaddlePortalSession();
    if (portalUrl != null) {
      await launchUrl(Uri.parse(portalUrl));
    }
    ```

### Entitlement identity

For in-app purchases, iOS usually handles entitlement recovery through the user's Apple ID. For Paddle App2Web, Helium uses the user ID you provide, plus Apple ID context where available.

| Identity path | What to configure |
| :-- | :-- |
| Users have a stable user ID before purchase | Set the user ID before showing the paywall. This is the cleanest path. |
| Users can buy before signing in | Allow checkout without a user ID, then set the stable user ID as soon as the user signs in. Native iOS uses `Helium.config.allowWebCheckoutWithoutUserId = true`. |
| The user ID may change after purchase | Send the updated stable user ID to Helium and test that entitlement recovery still works. |
| You never have a stable user ID | Helium can still use Apple ID context where available, but support, recovery, and cross-device behavior are weaker. |

<Warning>
  Treat the return URL as a reason to refresh entitlement state, not as proof of purchase. Unlock access only after Helium reports an active entitlement.
</Warning>

<Accordion title="Optional: Apple Pay capability">
  Add Apple Pay capability only if your rollout uses Apple Pay availability for targeting or fallback logic.

  For native iOS, open your project settings in Xcode, go to **Signing & Capabilities**, and add **Apple Pay**. This requires an Apple [merchant identifier](https://developer.apple.com/help/account/capabilities/configure-apple-pay#create-a-merchant-identifier).

  For Expo, add the Apple Pay entitlement to `app.json` or `app.config.js`:

  ```json
  {
    "expo": {
      "ios": {
        "entitlements": {
          "com.apple.developer.in-app-payments": [
            "your-merchant-identifier"
          ]
        }
      }
    }
  }
  ```

  For Flutter, open `ios/Runner.xcodeproj` in Xcode, select the Runner target, go to **Signing & Capabilities**, and add **Apple Pay**.
</Accordion>

## Connect Paddle to Helium

After the SDK path is underway, connect Paddle to Helium so products can be synced and used in paywalls. This Paddle API key is for Helium's backend integration with Paddle, not the mobile SDK.

<Steps>
  <Step title="Create a Paddle API key for Helium">
    Create an API key in Paddle with the permissions Helium needs to sync products, create transactions, manage subscriptions, configure notifications, and open customer portal sessions.

    <Accordion title="Required Paddle API key permissions">
      | Permission | Why Helium needs it |
      | :-- | :-- |
      | `product.write` | Create and sync products. |
      | `price.write` | Create and sync prices. |
      | `customer.write` | Create customers for end users. |
      | `transaction.write` | Create transactions for purchases. |
      | `subscription.write` | Manage subscription lifecycle. |
      | `adjustment.read` | Read refunds and chargebacks. |
      | `notification_setting.write` | Configure notifications or webhooks. |
      | `payment_method.read` | Read saved payment methods where needed. |
      | `client_token.write` | Create client-side tokens. |
      | `discount.read` | Read discount information for paywalls. |
      | `customer_portal_session.write` | Create customer portal sessions for subscription management. |
    </Accordion>
  </Step>
  <Step title="Add the Paddle API key to Helium">
    In the Helium dashboard, go to Settings, then Integrations, then Paddle. Paste the API key into the Paddle integration.
  </Step>
  <Step title="Configure Paddle notifications">
    Helium will provide a notification or webhook destination. Add it in Paddle, then copy the signing secret back into Helium if prompted. This powers entitlement updates and revenue reporting.
  </Step>
  <Step title="Sync products and prices">
    Sync Paddle products into Helium after setup. Sync again after product, price, trial, or discount changes.
  </Step>
</Steps>

## Configure products, trials, and discounts

Paddle catalog setup matters because Helium uses products and prices to decide which offers to show and which entitlements to grant.

### Products and prices

Create the Paddle products and prices you plan to sell, then sync them into Helium. If you use trials or discounts, create the offer variants before building the paywall.

### Trials

Helium's trial model is trial/no-trial pairing.

For each price that needs a trial:

1. Create a trial Paddle price.
2. Create an equivalent no-trial Paddle price.
3. Link the trial and no-trial versions in Helium.
4. Helium routes eligible users to the trial version and ineligible users to the no-trial version.

Helium uses this pairing to replicate iOS intro-offer behavior for Paddle. A person who has already used a trial will not see the trial version again.

Paddle trials require a payment method up front. Payment is deferred until the trial converts.

<Warning>
  Do not rely on a raw Paddle trial alone unless Helium has confirmed that this is the intended behavior for your launch. A raw trial can behave differently from Helium's trial eligibility logic.
</Warning>

### Discounts

Set up Paddle discounts in Helium. Today, Helium supports new-user acquisition offers: once someone has used a trial or discount, Helium stops showing them trial/discount variants for now.

To create a discounted variant of a product:

1. Start from the Helium product bucket that uses the base Paddle price.
2. Create a separate product bucket for the discounted variant.
3. Reuse the same Paddle price and apply the Paddle discount in Helium.

You can reuse the same Paddle price in multiple Helium product buckets and apply different discounts to create different offer variants.

Helium generally supports Paddle discounts except discount codes. Multi-step offers, such as trial plus discount or multiple discount tiers, do not display correctly today.

Intro pricing such as `$1.99 for the first 3 months, then $9.99/month` is not supported as a multi-step offer today.

We plan to add more flexible offer types shortly.

## Build the App2Web paywall

Start from an in-app paywall that already works with normal iOS in-app purchases. App2Web converts that paywall into a forked purchase experience: eligible users can move to web checkout through Paddle, while users who should stay in-app can continue through the normal IAP path.

For iOS, the in-app path still matters. Apple's external purchase rules can vary by region and app category, but maintaining support for in-app purchases is the consistent requirement to design around. In Helium, that means your App2Web experience should preserve a working IAP route instead of replacing every iOS purchase path with Paddle.

### How the fork works

Paddle App2Web uses two paywalls:

| Paywall | Where it appears | What it does |
| :-- | :-- | :-- |
| In-app App2Web paywall | Inside your app | Shows the offer and gives eligible users a web checkout path plus an in-app purchase path. |
| Hosted web paywall | In the browser | Presents the web checkout destination and Paddle purchase flow. |

The in-app paywall is the starting point. It is targeted through workflows, experiments, and audiences.

To start a web checkout, configure the in-app paywall CTA to purchase a Paddle product. That purchase event is what tells Helium to open the hosted web paywall and Paddle checkout.

The in-app App2Web paywall must be linked to a hosted web paywall before it can be published.

If the user chooses the in-app path, show the normal iOS IAP purchase path. If the user goes to web checkout and returns without purchasing, show the in-app purchase option quickly instead of sending them back to web checkout again.

### Convert the baseline paywall

Before converting the paywall, confirm:

- The baseline paywall is approved.
- Product display and pricing are correct.
- Trial behavior is understood.
- The normal in-app purchase path has passed QA.
- Fallback behavior has been considered for the app.

Then adapt the baseline:

1. Keep the first screen close to the approved baseline paywall.
2. Add a Paddle web checkout path for eligible users.
3. Add a clear in-app purchase path, usually labeled `Continue in-app`.
4. Configure the web checkout CTA to purchase the Paddle product.
5. Configure `Continue in-app` to show the normal iOS IAP paywall or product.

Ideally, this is a multi-step paywall. The first step can present the offer and point eligible users toward web checkout. A later step can preserve the in-app purchase path for users who should stay in-app or who return from web checkout without buying.

### Product configuration rules

Keep the product paths clean:

- The Paddle hosted web paywall should contain Paddle products only.
- The in-app App2Web paywall should contain the Paddle product used to start web checkout.
- The in-app purchase path should use iOS IAP products only.
- `Continue in-app` should show the normal iOS purchase path, not a generic next step.
- Android products should stay out of the iOS App2Web flow.

If your current product bucket combines iOS and Android products, create an iOS-only version for the App2Web path.

<Accordion title="How to configure your products">
  A normal paywall may use one product bucket for multiple platforms. App2Web works better when the web and in-app purchase paths are explicit.

  Setup:

  1. Start from the product setup used by the baseline IAP paywall.
  2. Create or select a Paddle product bucket for the web checkout CTA.
  3. Create or select an iOS-only product bucket for the `Continue in-app` path.
  4. Configure the primary web CTA to purchase the Paddle product.
  5. Configure `Continue in-app` to show the normal iOS IAP product or paywall.
  6. Keep Android products out of the iOS App2Web path.

  Confirm the exact dashboard steps with Helium if your current bucket structure is complex.
</Accordion>

### Build the in-app App2Web paywall

The in-app App2Web paywall is where the purchase path is chosen.

If you need to block prepaid cards, configure that on the App2Web paywall.

Checklist:

- Primary web CTA is configured to purchase the Paddle product.
- `Continue in-app` shows the normal iOS IAP purchase path.
- Pricing, trial copy, plan names, and billing periods match actual Paddle configuration.
- The paywall does not imply that checkout is complete before the user pays.
- The user can recover if browser checkout is canceled or fails.
- Users who should stay on IAP can still purchase in-app.

### Tips for the in-app flow

- Use a multi-step paywall when possible. It gives you room to send eligible users to web checkout while still preserving the in-app path.
- Keep the primary CTA placement stable across steps. If you want more users to convert on web, multiple sequential CTAs in the same location can make the web path feel like the natural next action.
- After a user returns from web checkout without purchasing, move them quickly to the in-app purchase option instead of sending them back to web checkout again.
- Avoid changing the offer, price framing, and purchase path all at once. Start from the baseline paywall so testing isolates the App2Web change.

### Build the hosted web paywall

The hosted web paywall is the browser destination for Paddle checkout. It is served from `clickthrough.to` and exists to back the linked in-app App2Web paywall.

The hosted web paywall can be customized independently from the in-app paywall. For a first launch, keep the offer aligned unless you are intentionally testing a different checkout experience, such as an upsell.

Both the in-app App2Web paywall and the linked hosted web paywall must be published before the flow works end to end.

Checklist:

- It contains Paddle products only.
- It is linked to the in-app App2Web paywall.
- It is published before end-to-end testing.
- The CTA starts Paddle checkout.
- Copy and pricing match the intended checkout offer.
- The layout works in a mobile browser.
- Localizations match the in-app experience, if your app supports multiple languages.
- Dismissal/back behavior is intentional for the browser handoff.
- Success return opens the app.
- Cancel or failure return opens the app and does not unlock the user.
- Entitlement refresh has been tested after return.

Limitations to keep in mind:

- Today, hosted web paywalls are less capable than in-app Helium paywalls.
- App-only variables, payloads, and localized app context may not be available after the browser handoff.
- Targeting and audience logic should happen before the user is sent to the hosted web paywall.

## RevenueCat setup

If your app uses RevenueCat, associate Paddle purchases with the same RevenueCat App User ID that your app expects.

RevenueCat has its own customer identity called App User ID. Helium, Paddle, and RevenueCat need to agree on which user made the purchase.

### RevenueCat configuration

Follow RevenueCat's Paddle documentation and confirm the exact setup with Helium for your app.

Typical setup:

1. Open RevenueCat.
2. Go to your project.
3. Add a Paddle web configuration.
4. Add the required Paddle API key or connection details.
5. Use automatic purchase tracking unless Helium tells you otherwise.
6. Configure App User ID matching with a custom field, typically `rc_user_id`.
7. Import Paddle products into RevenueCat.
8. Attach Paddle products to the correct RevenueCat entitlement.

Useful RevenueCat references:

- [RevenueCat Paddle Billing](https://www.revenuecat.com/docs/web/integrations/paddle)
- [RevenueCat Paddle app-to-web purchases](https://www.revenuecat.com/docs/web/guides/paddle-app-to-web)
- [RevenueCat Paddle server notifications](https://www.revenuecat.com/docs/platform-resources/server-notifications/paddle-server-notifications)

<Note>
  For the fastest post-purchase app experience, check Helium entitlement state immediately after the user returns from Paddle. RevenueCat may be slightly delayed because it depends on server notifications and identity matching.
</Note>

## Target and launch App2Web

Target the in-app App2Web paywall through workflows, experiments, and audiences. Do not target the hosted web paywall directly; it should only be reached through the linked App2Web flow.

Launch as a test audience first, then an experiment or controlled rollout. Before expanding traffic, confirm:

| Area | Confirm |
| :-- | :-- |
| SDK | The app build includes external web checkout support and passes returned URLs to Helium. |
| Return URLs | Success and cancel URLs open the app in cold start, backgrounded, and already-open states. |
| Identity | User ID, checkout-before-sign-in, logout, and account-switching behavior match your plan. |
| Products | Paddle products are used for web checkout; iOS IAP products remain available for the in-app path. |
| Offers | Trial/no-trial pairing and discount behavior are correct in Helium. |
| Paywalls | The in-app App2Web paywall and hosted web paywall are linked and published. |
| Targeting | Only eligible users see App2Web; users who should stay on IAP can still purchase in-app. |
| RevenueCat | If used, the Paddle purchase maps to the expected RevenueCat App User ID and entitlement. |
| Paddle live setup | Paddle live access, live credentials, and live notifications are configured. |
| Compliance | Supported storefronts, Paddle requirements, and iOS IAP fallback are reviewed before rollout. |

Test the full path with live payments before launch: in-app paywall, Paddle checkout, success return, cancel return, failed payment, entitlement refresh, subscription management, and rollback.

<Warning>
  External purchase rules differ by region and app category. Review Apple's current guidance before exposing App2Web broadly.
</Warning>

Useful references:

- [Apple App Review Guidelines](https://developer.apple.com/app-store/review/guidelines/)
- [Apple guideline update on external purchase links](https://developer.apple.com/news/?id=9txfddzf)
- [Paddle go-live checklist](https://developer.paddle.com/build/go-live-checklist)

After launch, watch checkout start, checkout completion, return URL failures, entitlement grant failures, RevenueCat sync lag, Paddle webhook failures, and support tickets about access or cancellation. If something looks wrong, pause the App2Web audience or experiment and keep the normal IAP path available.

## Troubleshooting

| Problem | Likely cause | First check |
| :-- | :-- | :-- |
| Checkout does not return to app | Deep link or universal link is not configured correctly | Open the success and cancel URLs directly on a device. |
| Purchase succeeds but app stays locked | Entitlement refresh or identity mapping issue | Check Helium entitlement state for the user ID. |
| RevenueCat does not show purchase | App User ID matching or Paddle notification issue | Check `rc_user_id`, RevenueCat Paddle config, and server notifications. |
| User sees Paddle when they should not | Targeting/audience issue | Check workflow, experiment, and audience rules. |
| User sees iOS product in web path | Product bucket setup issue | Confirm hosted web paywall uses Paddle-only products. |
| Hosted web paywall cannot publish | Missing or invalid product/linking state | Confirm the web paywall has valid Paddle products and is linked correctly. |
| Trial appears for an ineligible user | Trial/no-trial mapping issue | Confirm Helium trial pairing and eligibility logic. |
| Pre-live checks pass but live purchase fails | Live account/catalog/credential mismatch | Confirm Paddle account status, live products, live API key, and live notifications. |
| Manage subscription opens wrong destination | App is routing Paddle subscribers to App Store settings | Use the Paddle portal session for Paddle subscribers. |
| Checkout fails for some users | Payment method, region, compliance, or Paddle account issue | Check Paddle transaction details and eligibility targeting. |
